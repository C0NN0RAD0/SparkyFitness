# AI Integration Analysis

This document provides a comprehensive analysis of the AI integrations in SparkyFitness, covering supported providers, the system prompts and instructions given to LLMs, and the tools available to the AI for controlling application data via MCP.

## Overview

SparkyFitness integrates AI through two distinct pathways:

1. **Inline Chat AI** — A direct LLM call made from the backend (`SparkyFitnessServer/services/chatService.js`) when a user sends a message to Sparky. The AI parses natural language into structured intents, and the application handles the actual database operations.

2. **MCP Server** — A separate Model Context Protocol server (`SparkyFitnessMCP/`) that exposes structured tools for AI agents (e.g., n8n workflows, Claude Desktop, or similar MCP-compatible clients) to directly read and write user data.

## 1. Supported AI Providers

All provider configuration is in `SparkyFitnessServer/ai/config.js`.

| Provider | `service_type` | Default Chat Model | Default Vision Model |
|---|---|---|---|
| OpenAI | `openai` | `gpt-4o-mini` | `gpt-4o-mini` |
| Anthropic | `anthropic` | `claude-3-5-sonnet-20241022` | `claude-3-5-sonnet-20241022` |
| Google Gemini | `google` | `gemini-pro` | `gemini-2.5-flash` |
| Mistral | `mistral` | `mistral-large-latest` | `pixtral-large-latest` |
| Groq | `groq` | `llama3-8b-8192` | `llama-3.2-11b-vision-preview` |
| OpenRouter | `openrouter` | `google/gemma-3-27b-it:free` | `google/gemini-2.5-flash` |
| Ollama (local) | `ollama` | user-configured | `llava` |
| OpenAI-compatible | `openai_compatible` | user-configured | user-configured |

Provider settings are stored in the `public.ai_service_settings` database table with API keys encrypted at rest using **AES-256-GCM** (`SparkyFitnessServer/security/encryption.js`).

### Provider Selection Priority

When a user sends a chat message:
1. The frontend passes a `service_config_id` (UUID) identifying the chosen AI service
2. The backend fetches the service settings, decrypts the API key, and routes the request to the correct provider

If no user-specific setting is active, the system falls back to the admin-configured global setting.

## 2. System Prompt

**Location:** `SparkyFitnessServer/services/chatService.js` — `processChatMessage()` function

The system prompt is constructed dynamically on every request. Here is the full prompt template:

```
You are Sparky, an AI nutrition and wellness coach. Your primary goal is to help users
track their food, exercise, and measurements, and provide helpful advice and motivation
based on their data and general health knowledge.

The current date is {YYYY-MM-DD}.

**CRITICAL INSTRUCTION:** When the user mentions "water" in any context related to
consumption or intake, you MUST use the 'log_water' intent. Do NOT classify water as
a 'log_food' item.

You will receive user input, which can include text and/or images. Your task is to
identify the user's intent and extract relevant data. You MUST respond with a JSON
object containing the 'intent' and 'data', strictly adhering to the defined intents
and their required data structures.

For image inputs:
- Analyze the image to identify food items, estimate quantities, and infer nutritional
  information.
- If the image clearly shows food, prioritize the 'log_food' intent.
- Extract food_name, quantity, unit, and meal_type from the image content.
- **CRITICAL:** Always infer and include *estimated* nutritional details (calories,
  protein, carbs, fat, etc.) based on the identified food and estimated quantity,
  populating the corresponding fields in the 'log_food' intent's data. Do NOT default
  to 0 if an estimation can be made.
- If the image is not food-related or unclear, treat the text input as primary.

**IMPORTANT:** If the user specifies a date or time (e.g., "yesterday", "last Monday",
"at 7 PM"), extract this information and include it as a 'entryDate' field in the top
level of the JSON object. **Provide relative terms like "today", "yesterday",
"tomorrow", or a specific date in 'MM-DD' or 'YYYY-MM-DD' format. Do NOT try to
resolve relative terms to a full date yourself.** If no date is specified, omit the
'entryDate' field.

When the user mentions logging food, exercise, or measurements, prioritize extracting
the exact name of the item (food name, exercise name, measurement name) as accurately
as possible from the user's input. This is crucial for looking up existing items in
the database.

Here are the user's existing custom measurement categories:
{CUSTOM_CATEGORIES_LIST}

When the user mentions a custom measurement, compare it to the list above. If you
find a match or a very similar variation (considering synonyms and capitalization),
use the **exact name** from the list in the 'name' field of the measurement data.
If no clear match is found in the list, use the name as provided by the user.
```

### Dynamic Substitutions

| Placeholder | Source | Purpose |
|---|---|---|
| `{YYYY-MM-DD}` | `new Date().toISOString().split("T")[0]` | Current date for relative date context |
| `{CUSTOM_CATEGORIES_LIST}` | DB query: `measurementRepository.getCustomCategories(userId)` | User's custom measurement category names so the AI uses exact matches |

## 3. Intent System

The system prompt instructs the AI to classify every input into exactly one of six intents and respond with a JSON object:

### Intent: `log_food`

Used when the user wants to log a food or beverage (not water).

```json
{
  "intent": "log_food",
  "data": {
    "food_name": "string",
    "quantity": "number",
    "unit": "g | piece | oz | ml | cup | serving",
    "meal_type": "breakfast | lunch | dinner | snacks",
    "calories": "number (optional)",
    "protein": "number (optional)",
    "carbs": "number (optional)",
    "fat": "number (optional)",
    "saturated_fat": "number (optional)",
    "polyunsaturated_fat": "number (optional)",
    "monounsaturated_fat": "number (optional)",
    "trans_fat": "number (optional)",
    "cholesterol": "number (optional)",
    "sodium": "number (optional)",
    "potassium": "number (optional)",
    "dietary_fiber": "number (optional)",
    "sugars": "number (optional)",
    "vitamin_a": "number (optional)",
    "vitamin_c": "number (optional)",
    "calcium": "number (optional)",
    "iron": "number (optional)"
  },
  "entryDate": "today | yesterday | MM-DD | YYYY-MM-DD (optional)"
}
```

### Intent: `log_exercise`

Used when the user wants to log physical activity.

```json
{
  "intent": "log_exercise",
  "data": {
    "exercise_name": "string",
    "duration_minutes": "number | null",
    "distance": "number | null",
    "distance_unit": "miles | km | null"
  },
  "entryDate": "optional"
}
```

### Intent: `log_measurement`

Used for body metrics (weight, steps, body measurements, or custom metrics).

```json
{
  "intent": "log_measurement",
  "data": {
    "measurements": [
      {
        "type": "weight | neck | waist | hips | steps | custom",
        "value": "number",
        "unit": "kg | lbs | cm | inches | steps | null",
        "name": "string (required only when type is 'custom')"
      }
    ]
  },
  "entryDate": "optional"
}
```

### Intent: `log_water`

Used when the user mentions drinking water. **The system prompt explicitly prohibits classifying water as `log_food`.**

```json
{
  "intent": "log_water",
  "data": {
    "glasses_consumed": "number (default: 1)"
  },
  "entryDate": "optional"
}
```

### Intents: `ask_question` / `chat`

Used for general questions, advice, or casual conversation.

```json
{
  "intent": "ask_question",
  "data": {},
  "response": "string — the AI's conversational reply"
}
```

## 4. Conversation History

**Frontend** (`SparkyFitnessFrontend/src/api/Chatbot/sparkyChatService.ts`): The frontend sends the last 10 messages (both user and assistant) from the in-memory chat state to the backend on every request.

**Backend**: The backend prepends the system prompt and forwards all messages (user + assistant) to the AI provider. This provides full multi-turn context so the AI can understand follow-up questions and references to previous entries.

### Message Flow

```
Frontend                       Backend (chatService.js)              AI Provider
   │                                    │                                 │
   │── POST /chat ──────────────────────►│                                 │
   │   {messages: [...last 10],          │                                 │
   │    service_config_id: "uuid"}       │                                 │
   │                                    │── [system prompt]              │
   │                                    │   + [last 10 messages]  ────────►│
   │                                    │                                 │
   │                                    │◄── JSON intent response ────────│
   │◄── {content: "...JSON..."} ────────│                                 │
```

### Chat History Persistence

Messages are also saved to the `public.sparky_chat_history` database table, separate from what is sent to the AI. The saved history is loaded on page refresh to restore the visual chat log.

## 5. Food Options Generation

When the AI cannot find a food in the database, it triggers a separate AI call with a different prompt to generate 2-3 realistic nutrition estimates.

**Trigger:** `POST /chat/food-options` with `{ foodName, unit, service_config_id }`

**System Prompt (food options):**
```
You are Sparky, an AI nutrition and wellness coach. Your task is to generate minimum
3 realistic food options in JSON format when requested. Respond ONLY with a JSON array
of FoodOption objects, including detailed nutritional information (calories, protein,
carbs, fat, saturated_fat, ...). Always provide estimated nutritional details. Do NOT
include any other text.

CRITICAL: When a unit is specified in the request (e.g., 'GENERATE_FOOD_OPTIONS:apple
in piece'), ensure the `serving_unit` in the generated FoodOption objects matches the
requested unit exactly, if it's a common and logical unit for that food.
```

**User Message Format:** `GENERATE_FOOD_OPTIONS:{food_name} in {unit}`

**Expected JSON Response:**
```json
[
  {
    "name": "Apple (medium)",
    "calories": 95,
    "protein": 0.5,
    "carbs": 25,
    "fat": 0.3,
    "serving_size": 1,
    "serving_unit": "piece"
  }
]
```

## 6. Image Analysis

The inline chat AI supports multimodal input (images + text). When a user uploads a photo of food:

1. The frontend converts the image file to a base64 data URL
2. The data URL is sent as an `image_url` content part in the user message
3. The AI analyzes the image and returns a `log_food` intent with estimated nutrition

**Supported vision providers:** OpenAI (gpt-4o-mini), Anthropic (claude-3-5-sonnet), Google (gemini-2.5-flash), Mistral (pixtral-large), Groq (llama-3.2-vision), Ollama (llava).

**Note:** Ollama does not support multimodal input via the chat service — image data is silently dropped with a warning when an Ollama provider is selected.

## 7. MCP Server — Tools for AI Agents

**Location:** `SparkyFitnessMCP/src/`

The MCP (Model Context Protocol) server is a separate service running alongside the main backend. It exposes structured tools that MCP-compatible AI clients (e.g., Claude Desktop, n8n, Cursor, etc.) can call to directly read and write user data in the SparkyFitness database.

**Transport:** Streamable HTTP (stateless, one server instance per request)  
**Endpoint:** `POST /mcp`  
**Port:** `SPARKY_FITNESS_MCP_PORT` (default: `5435`)  
**Tool Discovery:** `GET /mcp/tools`

### 7.1 Nutrition Tools (`manage_food`)

The primary tool for nutrition tracking. Actions:

| Action | Description | Key Parameters |
|---|---|---|
| `search_food` | Search existing food items | `food_name`, `search_type` |
| `create_food` | Create a new custom food item | `food_name`, `macros` (full nutrition object) |
| `log_food` | Log a food entry to the diary | `food_name`, `quantity`, `unit`, `meal_type`, `entry_date` |
| `list_diary` | View food entries for a date | `entry_date` |
| `delete_entry` | Delete a food/meal diary entry | `entry_id`, `entry_type` |
| `update_entry` | Update a diary entry quantity | `entry_id`, `entry_type`, `quantity`, `unit` |
| `search_meal` | Search meal templates | `meal_name` |
| `log_meal` | Log a meal template to diary | `meal_id` or `meal_name`, `meal_type`, `entry_date` |
| `copy_from_yesterday` | Copy entries from a previous date | `source_date`, `target_date`, `meal_type` |
| `save_as_meal_template` | Save entries as a reusable meal | `entry_date`, `meal_type`, `meal_name` |

### 7.2 Exercise Tools (`manage_exercise`)

| Action | Description | Key Parameters |
|---|---|---|
| `search_exercises` | Search the exercise database | `exercise_name` |
| `create_exercise` | Create a custom exercise | `exercise_name`, `category`, `calories_per_hour` |
| `log_exercise` | Log a workout | `exercise_name`, `duration_minutes`, `entry_date`, `sets` |
| `list_exercise_diary` | View logged workouts | `entry_date` |
| `get_workout_presets` | List saved workout presets | — |
| `log_workout_preset` | Log a full workout preset | `preset_id`, `entry_date` |
| `delete_exercise_entry` | Remove a workout entry | `entry_id` |

### 7.3 Health Check-in Tools (`manage_checkin`)

| Action | Description | Key Parameters |
|---|---|---|
| `log_biometrics` | Log weight, steps, height, measurements | `weight`, `steps`, `height`, `neck`, `waist`, `hips` |
| `log_custom_metric` | Log a custom health metric | `category_name`, `value`, `unit` |
| `list_categories` | List user's custom metric categories | — |
| `create_category` | Create a new custom metric category | `name`, `measurement_type`, `frequency` |
| `log_mood` | Log a daily mood score | `mood_value` (1–10) |
| `log_fasting` | Track a fasting period | `fasting_status`, `entry_date` |
| `log_sleep` | Log sleep duration and quality | `duration_seconds`, `sleep_score` |
| `list_checkin_diary` | View health check-in history | `entry_date` |

### 7.4 Coaching / Analytics Tools

## 9. Security Considerations

### API Key Storage
- All AI provider API keys are encrypted at rest using **AES-256-GCM** with a 256-bit key
- The encryption key is read from the `SPARKY_FITNESS_API_ENCRYPTION_KEY` environment variable
- Encrypted keys, IVs, and GCM authentication tags are stored separately in the database
- API keys are never returned in API responses (stripped before returning settings to the frontend)

### Access Control
- All chat API endpoints require JWT authentication (`authenticate` middleware)
- The `service_config_id` passed by the client is validated against the authenticated user's own AI service settings, plus any global/public settings
- Users cannot access or modify another user's AI service configurations
- Only administrators can create, modify, or delete global (public) AI service settings

### MCP Server Authentication
- The MCP server connects directly to the PostgreSQL database using a hardcoded `MOCK_USER_ID` from configuration (see `SparkyFitnessMCP/src/config.ts`)
- This means any client with network access to the MCP port can perform data operations as that user
- **In production, the MCP server should be restricted to trusted networks and the `MOCK_USER_ID` should be set to a real user ID or replaced with proper auth**

### Prompt Injection
- User messages are passed directly to AI providers without sanitization
- Custom measurement category names fetched from the database are injected into the system prompt; these are user-controlled strings
- Malicious users could attempt to override the system prompt by including prompt-injection content in their messages or category names

### Data Sent to External AI Providers
The following data may be sent to the configured external AI provider:
- The user's custom measurement category names (injected into the system prompt)
- The full content of the user's chat messages (including any personally identifiable information the user chooses to share)
- Images uploaded by the user (base64-encoded)
- Conversation history (last 10 messages)

No other user data (e.g., food diary, weight history, goals) is sent to external AI providers by the inline chat integration. The MCP server reads data from the local database and returns it as tool call responses, which may then be included in AI context by the MCP client.

## 10. Provider-Specific Behaviour

### Google Gemini
- System prompt is sent via the `systemInstruction` field (separate from conversation `contents`)
- Markdown formatting characters (backticks, asterisks, etc.) are stripped from the system prompt as they may interfere with Gemini's instruction parsing
- System-role messages are excluded from `contents` since the instruction is handled by `systemInstruction`
- Image data is converted from OpenAI-style `image_url` format to Google's `inline_data` format

### Anthropic Claude
- System prompt is sent via the top-level `system` field (not as a message)
- System-role messages are excluded from the `messages` array
- `max_tokens` is capped at 1000 for chat responses

### Ollama (Local)
- Image data in messages is ignored with a warning (no multimodal support in the current integration)
- Connection timeout defaults to 20 minutes to support slow local hardware
- Supports optional API key authentication via Bearer token

### OpenRouter
- Sends additional headers: `HTTP-Referer: https://sparky-fitness.com` and `X-Title: Sparky Fitness`
